# v0.13.0 - 2026-03-27

## 前情与承接

`v0.12.1` 已经处理过一次 provider 兼容性补丁：当图片附件问答触发不兼容的 `messages[].content` 数组格式时，系统会自动降级重试。这说明当前接入的部分 OpenAI-compatible 供应商并不能稳定支持完整 chat payload。  
`v0.12.2` 主要整理了聊天工作台的前端结构与交互，但模型侧仍然以单轮请求为主，同会话上下文记忆和消息编辑边界还没有统一收口。  
`v0.13.0` 承接这两条线：一边补齐同会话连续对话记忆，一边继续把 provider 差异和错误页兼容处理前移到 `LLMService`。

## 版本概览

本版本聚焦聊天链路的连续对话能力与模型兼容性治理，在不改数据库结构的前提下补齐同会话上下文记忆，并收紧消息编辑边界，避免会话状态失真。

## 主要更新

1. 同一会话的最近 6 轮 `ask/echo` 历史会作为上下文重新注入模型，普通聊天与 RAG 路径均支持连续对话记忆。
2. 编辑与重新生成能力收敛为“仅允许当前会话最后一条非 `action` 消息”，前后端同时校验，避免中间消息改写导致后续上下文失效。
3. 新增 `LLM_HISTORY_MODE=auto | native | compat` 三档历史注入模式，默认 `auto`，兼容上游对多轮 `messages` 支持不稳定的模型。
4. 上游返回 HTML `502`/网关错误页时，会先被清洗为可读错误摘要，避免整页 HTML 直接渲染到聊天区。

## 交互与行为调整

1. 前端仅在最后一条可编辑消息上显示“编辑 / 重新生成”入口，历史消息不再暴露相同操作。
2. 后端编辑接口新增最新消息校验，绕过前端直接请求也会被拒绝。
3. 编辑最后一条消息时，只会带入该消息之前的会话历史，不会把它自己和其后的消息错误地回灌进新请求。
4. 当模型原生多轮请求失败时，`auto` 模式会自动降级为兼容模式重试，优先保证 Demo 场景可用性。

## 兼容性说明

1. 本次更新不涉及数据库表结构调整，继续复用现有 `chat_history` 记录完成上下文拼接。
2. 默认历史窗口为最近 6 轮，可通过配置项 `llm_history_turns` 调整。
3. 默认历史模式为 `auto`；若要严格验证模型原生多轮能力，可显式切换为 `native`；若优先追求稳定兼容，可切换为 `compat`。
4. 仅 `ask/echo` 类型消息会参与上下文记忆，`action` 类消息不会进入模型历史，避免工具输出污染对话。
