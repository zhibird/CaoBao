# v0.18.0 - 2026-04-21

## 前情与承接

`v0.18.0` 承接了 `v0.15.0` 的 PostgreSQL-first 运行路径、`v0.16.0` 的标准登录鉴权闭环，以及 `v0.17.0` 的企业工作台前端重构。

到这个阶段，项目的主链路能力已经比较完整，但还存在两类明显缺口：

1. 虽然 `docker-compose.yml` 和 `Dockerfile` 已经存在，但容器运行链路还不够闭环，尤其是卷权限、运行身份、健康检查、OCR 依赖和发布说明仍然不够扎实。
2. 最近一轮前端工作台整理后，首页 `HEAD /` 请求、折叠 rail 的可访问性语义，以及设置面板里的 Admin 承接方式都发生了变化，需要同步补齐测试与细节修复。
3. 登录鉴权与 PostgreSQL 路径已经进入稳定期后，一些历史兼容问题也更容易暴露出来，比如 PostgreSQL 布尔默认值不应继续沿用 `default 1` 的旧写法。

这次版本的目标，就是把 Docker 运行与发布说明真正补齐，同时把近期这批边界 bug 一并收口，为 `v0.18.0` tag 提供更可直接发布的状态。

## 版本概览

本版本主要完成：

1. 补齐 Docker 运行链路，包括非 root 运行、启动迁移、健康检查、OCR 依赖、卷权限和 `.dockerignore` 收口。
2. 修复 PostgreSQL 布尔默认值兼容、首页 `HEAD /` 请求，以及工作台 rail / 设置面板相关的前端细节问题。
3. 补齐 `README` 的 Docker 使用说明、版本号默认值和 `v0.18.0` 的版本日志。

## 主要更新

1. `Dockerfile` 增加 `gosu`、`tesseract-ocr`、`tesseract-ocr-chi-sim`、健康检查和运行时目录准备逻辑。
2. `docker-compose.yml` 补齐应用环境变量透传、`init: true`、本机回环端口绑定和 API 健康检查。
3. `scripts/docker-entrypoint.sh` 改为先修正 `/data` 卷权限，再以 `caibao` 用户执行 Alembic 与 Uvicorn。
4. 首页 `/` 改为同时支持 `GET` 与 `HEAD`，避免健康检查或代理探测静态首页时直接返回不一致结果。
5. 用户模型与 Alembic 迁移中的 `is_active` 默认值统一改为 `sa.true()` / `true()`，消除 PostgreSQL 下的布尔默认值兼容隐患。
6. 工作台左 rail 的折叠逻辑增加 `aria-hidden` 与 `inert` 语义，设置面板也重新整理了账号、模型和高级设置的布局承接。
7. `README` 新增更完整的 Docker 说明，明确首次启动行为、端口绑定策略、常用命令与发布建议。

## 问题现象

1. Docker 虽然能构建，但 volume 初始化后应用用户不一定对 `/data/uploads` 有写权限，附件上传链路存在风险。
2. 首页只支持 `GET /` 时，某些 `HEAD /` 探测路径会返回不符合预期的结果。
3. PostgreSQL 迁移和 ORM 模型里继续使用 `default 1` 风格的布尔默认值，不够稳妥，也不利于跨方言行为保持一致。
4. 左 rail 收起后虽然视觉上隐藏，但缺少更明确的可访问性状态同步；同时设置面板里的 Admin 承接位置也需要更统一的整理。

## 根因分析

1. 容器卷的所有权和镜像内目录所有权不是同一回事，单纯在 build 阶段 `chown` 不能覆盖运行时新挂载的 Docker volume。
2. 首页静态文件返回逻辑最初只按普通页面访问设计，没有把 `HEAD` 请求也纳入同一条返回路径。
3. 历史兼容写法更偏向 SQLite 语义，但当前主库已经切换到 PostgreSQL，布尔默认值应该回到更标准的 `true` 表达。
4. 前端工作台在快速演进过程中，视觉状态和 DOM / 无障碍语义并没有每次同步补齐，需要靠补充测试把这些细节锁住。

## 修复内容

1. Docker 入口脚本补上 `/data/uploads` 目录准备与权限修正，并通过 `gosu caibao` 启动迁移和 API 进程。
2. 镜像与 Compose 同时补上健康检查，让容器状态、启动顺序和运行时观测都更稳定。
3. 首页静态入口改为 `@app.api_route("/", methods=["GET", "HEAD"])`，统一处理页面访问与探测请求。
4. `users.is_active` 和对应 Alembic 迁移统一改为 `server_default=true()` / `sa.true()`。
5. rail 折叠时同步设置 `aria-expanded`、`aria-hidden` 与 `inert`，减少“视觉隐藏但仍可聚焦”的状态偏差。
6. 设置面板把 Admin 承接收回到高级设置区，避免工作台主导航继续暴露开发者入口。
7. `README`、`.env.example`、默认 `APP_VERSION` 与 changelog 一并更新到 `v0.18.0`。

## API / 数据模型 / 配置变更

1. `Settings.app_version` 默认值更新为 `0.18.0`。
2. `.env.example` 与 `docker-compose.yml` 的默认 `APP_VERSION` 同步到 `0.18.0`。
3. `Dockerfile`、`docker-compose.yml` 与 `scripts/docker-entrypoint.sh` 共同形成更完整的容器启动链路。
4. 用户模型与 Alembic 迁移里的布尔默认值表达改为 PostgreSQL 友好的写法。

## 兼容性说明

1. Docker Compose 端口当前默认只绑定 `127.0.0.1`，更适合本机开发与单机部署；若需要对外暴露，需要显式调整端口绑定策略。
2. 容器启动时会执行 `alembic upgrade head`，因此正式环境应确保数据库连接与账号权限已提前配置好。
3. OCR 依赖已进入镜像，但是否真正启用仍取决于上传内容类型与业务链路是否走到图片解析路径。
4. 设置面板中的 Admin 入口位置发生调整，但 `Admin` 页面本身仍然保留在 `/web/admin.html`。

## 验证结果

1. `pytest -q` 通过，结果为 `178 passed, 1 skipped`。
2. `docker compose up --build -d` 可成功启动 `postgres` 与 `caibao-api`。
3. `Invoke-WebRequest http://localhost:8000/api/v1/health` 返回成功。
4. 容器内实测 `/data/uploads` 可写，且 `uvicorn` 进程实际以 `uid 1001` 运行。
5. 容器内 `tesseract --version` 可正常输出，说明 OCR 依赖已安装到镜像。
