# Changelog Writing Guide

## 历史报告归纳

对现有 `changelogs/*.md` 做过一轮归纳后，可以得到几个比较稳定的事实：

1. `版本概览` 是最稳定的一级栏目，几乎所有版本都会保留。
2. `验证结果` 是第二稳定栏目，说明历史报告一直把“如何证明这次改动成立”当成核心输出。
3. `主要变更 / 主要更新 / 已实现 / 涉及文件 / 兼容性说明` 是高频辅助栏目，用于解释影响范围和落点。
4. 从 `0.10.x` 开始，复杂改动不再只写“改了什么”，而会补充“问题现象 / 根因 / 修复链路 / 数据模型 / API / 版本元数据”等专题段。
5. 到 `0.12.x`，写法又回到相对精炼，但仍然保留“按主题扩展”的习惯，例如 UI 版本会单开“交互与 UI 调整”，兼容性版本会强调 provider 边界。

## 历史报告的 4 种常见类型

### 1. 基线交付型

代表版本：

1. `0.2.0`
2. `0.12.0`

特征：

1. 适合“新增完整能力”或“一个阶段交付完成”的版本。
2. 结构通常是：`版本概览 -> 新增内容 -> 主要变更 -> 兼容性说明 -> 验证结果 -> 已实现`。
3. 强调“本版本交付了什么”，而不是深入问题排查过程。

### 2. 问题修复型

代表版本：

1. `0.10.1`
2. `0.10.4`
3. `0.12.1`

特征：

1. 适合 patch 版本、兼容性修复、线上 bug 修复。
2. 重点不是功能清单，而是：`问题现象 -> 根因 -> 修复内容 -> 验证结果`。
3. 如果问题来自上一版本，应明确写出“承接哪次改动、补哪个边界”。

### 3. 链路说明型

代表版本：

1. `0.10.0`
2. `0.10.3`
3. `0.11.1`

特征：

1. 适合上传、解析、索引、生成、多模态这类链路长、环节多的主题。
2. 需要把处理流程拆开说明，避免只写“功能已支持”。
3. 常见栏目是：`处理链路 / 正确流程 / API 变更 / 数据模型与迁移 / 依赖变更`。

### 4. 交互整理型

代表版本：

1. `0.11.2`
2. `0.12.2`

特征：

1. 适合前端结构重排、交互统一、预览体验优化。
2. 常见栏目是：`主要更新 -> 交互与 UI 调整 -> 兼容性说明 -> 验证结果`。
3. 重点是“用户实际体验发生了什么变化”，不是单纯的样式清单。

## 写作原则

1. 不要把 changelog 当成一次性提交说明，而要当成“本版本在产品演进中的定位说明”。
2. 写之前至少看 3 份历史报告：
   - 当前版本前一个版本；
   - 最近一个同模块或同链路版本；
   - 最近一个同类型版本（功能 / 修复 / UI / 链路）。
3. 第一段必须回答“本版本承接了什么”。不要让报告像孤立事件。
4. 如果本次是补丁版本，优先写清楚上一版本已经完成什么、遗留了什么、本次补上了什么。
5. 如果本次改动涉及 provider 兼容、数据迁移、前端缓存、版本元数据、接口边界，必须明确写出影响范围和不影响范围。
6. 文件列表只作为落点索引，不代替正文。正文必须先讲清“为什么改、改到了哪里、结果是什么”。

## 推荐写作流程

1. 先判断这次版本属于哪一类：基线交付、问题修复、链路说明、交互整理。
2. 回看相关历史版本，提炼“已完成 / 遗留问题 / 本次闭环”的三句话。
3. 先写 `前情与承接` 和 `版本概览`，再写专题段，不要反过来。
4. 只保留和这次主题强相关的专题段；不用为了整齐硬塞无关栏目。
5. 最后补 `验证结果` 与 `涉及文件 / 已实现`，确保读者能快速定位代码落点。

## 推荐的基线骨架

无论哪一类版本，建议都优先保留这几个部分：

1. `前情与承接`
2. `版本概览`
3. `主要更新 / 主要变更`
4. 一个或多个专题段
5. `验证结果`
6. `涉及文件` 或 `已实现`

## 专题段选择规则

### 出现 bug / 回归 / 兼容问题时

优先加：

1. `问题现象`
2. `根因分析` 或 `问题根因`
3. `修复内容` 或 `本次修复`

### 出现复杂链路改造时

优先加：

1. `处理链路`
2. `流程说明`
3. `API 变更`
4. `数据模型与迁移`
5. `依赖变更`

### 出现前端整理时

优先加：

1. `交互与 UI 调整`
2. `兼容性说明`
3. `版本与静态资源` 或缓存说明

### 出现策略性设计但未完全落地时

可选加：

1. `未落地提案`
2. `后续建议`

## 不建议的写法

1. 只看最新一个 changelog 就开始写，导致上下文断裂。
2. 只罗列文件，不解释版本在整体演进里的作用。
3. 所有版本都机械套同一个模板，不区分“功能版”和“修复版”。
4. 只写“做了什么”，不写“为什么是现在做、解决了上一版什么遗留问题”。
