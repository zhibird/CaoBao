# v0.11.1 - 2026-03-25

## 版本概览

`v0.11.1` 是对 `v0.11.0` “支持模型输出图片”能力的补丁修复版本。

`v0.11.0` 已经打通了“模型返回图片 -> 后端保存 `content_parts` -> 前端聊天区展示图片”的主链路，但在接入真实模型时又暴露出 4 类问题：

1. 图片型回复在分段续写时，会被错误注入占位文本 `Image output`，导致后续续写上下文被污染。
2. 模型返回的远程图片 URL 被直接写入历史记录，签名过期后，历史回放会失效。
3. 图片生成请求比普通文本回答更慢，超时后前端又自动回退到 Echo，用户看到的是“假成功”的 mock 回复。
4. 有些模型并不返回结构化 image block，而是直接输出 Markdown 图片文本，例如 `![Generated Image](data:image/...)`；旧逻辑会把它当普通文本原样展示。

本次修复的目标不是增加新能力，而是把“图片输出”这条链路补齐到可稳定使用的状态。

## 前因后果

### 问题 1：续写阶段被占位文本污染

#### 现象

当模型输出图片且 `finish_reason="length"` 时，后端会把当前分段先转成字符串 `Image output`，再作为 assistant 上下文继续发给模型。

这样会带来两个直接后果：

1. 最终 `answer` 被拼出类似 `Image outputThen explanation.` 的脏文本。
2. 下一轮续写看到的并不是“上一段真实图片输出”，而是一段伪造的字符串，容易让模型续写跑偏。

#### 根因

旧实现把“最终兜底展示文案”和“分段解析结果”混在了一起。

本来 `Image output` 只应该是最后没有文本时的兜底答案，但旧逻辑在 `_parse_llm_answer()` 阶段就提前生成了这个字符串，导致它参与了后续分段拼接和 continuation prompt。

#### 修复

文件：`app/services/llm_service.py`

- `_parse_llm_answer()` 不再在分段解析阶段生成 `Image output`。
- 只有整个请求结束后，如果最终没有文本但确实有图片，才在最外层生成一次兜底答案。
- 对于“只有图片且本段已被截断”的情况，后端现在停止在安全边界，不再带着伪造文本继续请求后续分段。

#### 修复后的效果

- 不会再把 `Image output` 混进真实回答文本里。
- 不会再把伪造字符串送回模型当作上一轮 assistant 输出。
- 纯图片回复仍然可以通过 `content_parts` 正常展示，`answer` 只作为兼容旧链路的兜底字段保留。

### 问题 2：远程图片 URL 直接入库，历史回放不稳定

#### 现象

部分模型返回的不是 `data:image/...`，而是带签名或短时有效的远程 URL。

旧逻辑会把这个 URL 原样写入 `content_parts`，再通过 `/chat/ask` 写进 `response_payload`。当用户刷新页面或之后回看历史时，这个远程 URL 可能已经失效，于是历史里的图片就打不开了。

#### 根因

旧实现只做“结构化提取”，没有做“持久化”。

也就是说，后端知道这是图片，但没有把图片内容真正保存下来，只是保存了一个外部地址。

#### 修复

文件：`app/services/llm_service.py`

- 当模型返回 `http(s)` 图片 URL 时，后端会立即尝试拉取图片二进制。
- 如果拉取成功且内容可识别为图片，则立即转成稳定的 `data:` URL。
- MIME 类型会按以下顺序推断：
  1. 响应头 `Content-Type`
  2. 文件签名
  3. URL 后缀
- 转换后的 `data:` URL 再进入 `content_parts`，再由 `/chat/ask` 持久化到历史记录里。

#### 修复后的效果

- 历史记录不再依赖临时签名 URL。
- 页面刷新、历史重放、重新打开会话时，图片都还能正常显示。

### 问题 3：图片生成超时后被错误伪装成 Echo

#### 现象

图片生成通常比普通文本回答慢。旧逻辑里，一旦上游模型超时或请求失败，前端会命中 `tryEchoFallback()`，直接改为调用 `/chat/echo`。

用户表面上看到了“有回复”，但这其实不是模型生成的结果，而是离线 Echo 兜底，属于典型的“假成功”。

#### 根因

问题同时存在于两层：

1. 后端默认沿用普通文本回答的超时时间，不够覆盖较慢的图片生成请求。
2. 前端的 Echo 兜底条件过宽，把“真实模型失败”也吞掉了。

#### 修复

文件：`app/services/llm_service.py`、`app/web/app.js`

- 后端会识别“疑似生成图片”的请求，并为这类请求使用更长的超时窗口。
- 前端不再把图片生成类请求自动改写成 Echo。
- 即使上游模型失败，这类请求现在也会把真实错误暴露给用户，而不是伪装成离线 mock 回复。

#### 修复后的效果

- 图片生成请求更容易在合理时间内成功完成。
- 如果失败，用户看到的是“真实失败”，而不是“假的成功”。
- 问题定位更直接，避免把模型失败误认为前端已经成功返回了图片。

### 问题 4：Markdown 图片被当成普通文本显示

#### 现象

部分模型输出图片时，不返回结构化 image block，而是直接返回一段文本：

`![Generated Image](data:image/jpeg;base64,...)`

旧逻辑会把这段内容完整塞进 `text` part，前端自然只能把它渲染成普通字符串，于是用户会在聊天区看到整段 base64 文本。

#### 根因

旧实现只会处理两类图片：

1. 结构化的 `image_url / output_image / generated_image`
2. 独立字段里的 `url / b64_json`

但它不会从纯文本里再二次识别 Markdown 图片语法。

#### 修复

文件：`app/services/llm_service.py`

- 后端新增了对 Markdown 图片语法的扫描。
- 无论是 `message.content` 直接是一段字符串，还是 `{"type":"text","text":"..."}` 这种文本 item，都会先拆分其中的 Markdown 图片。
- 识别到 `![alt](url)` 后，会转成真正的 image content part。
- 如果 URL 是远程地址，仍然会继续走“远程图片转 `data:` URL”的持久化逻辑。

#### 修复后的效果

- 类似 `![Generated Image](data:image/...)` 的输出不再原样展示。
- 前端拿到的是标准化后的 `content_parts`，因此会直接渲染为图片，而不是渲染成原始 Markdown 文本。

## 本次处理链路

这次补丁把“模型输出图片”的处理链路梳理成下面这条稳定流程：

1. 模型返回聊天响应。
2. 后端 `LLMService` 解析响应：
   - 先提取结构化 text/image part
   - 再从纯文本里补充识别 Markdown 图片
   - 对远程图片做即时拉取并转成稳定 `data:` URL
3. 后端组装 `LLMAnswer`：
   - `answer` 作为兼容旧接口的文本兜底字段
   - `content_parts` 作为真正的富内容承载字段
4. `RagChatService` 把 `content_parts` 写进 `ChatAskResponse`
5. `/chat/ask` 把完整响应持久化到 chat history 的 `response_payload`
6. 前端读取 `content_parts`：
   - `text` 按文本块渲染
   - `image` 按图片块渲染
7. 页面刷新或历史回放时，仍然从 `response_payload.content_parts` 还原完整内容

也就是说，图片展示现在不再依赖：

- 临时 URL 仍然有效
- 模型必须返回结构化 image block
- 前端去猜测 Markdown 文本怎么渲染

而是统一收敛到后端先标准化，再由前端按统一协议渲染。
