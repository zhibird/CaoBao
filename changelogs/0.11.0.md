# v0.11.0 - 2026-03-25

## 版本概览

本版本完成了“模型输出图片”的前后端主链路改造。此前菜包只支持：

1. 将图片作为输入附件发送给模型
2. 预览用户上传的图片/PDF 文件

但不支持：

1. 后端保留模型返回的图片块
2. 聊天接口把图片型回答返回给前端
3. 前端在聊天消息区渲染模型生成的图片

`v0.11.0` 之后，菜包已支持承接并展示 OpenAI-compatible 聊天响应中的“文本块 + 图片块”混合内容。

## 本次新增能力

### 1. 后端支持结构化解析模型输出内容

`app/services/llm_service.py` 从“只返回纯文本答案”升级为“返回文本兜底 + 结构化内容块”：

1. 支持解析普通文本内容
2. 支持解析内容列表中的文本块
3. 支持解析图片块
4. 支持图片 `url`
5. 支持 `data:image/...;base64,...`
6. 支持常见的 `b64_json / image_url / url` 结构

当模型输出图片时，系统不会再把图片部分丢掉。

### 2. 聊天响应新增 `content_parts`

`app/schemas/chat.py` 中的 `ChatAskResponse` 新增 `content_parts` 字段，用于承载结构化消息内容：

1. `type=text`
2. `type=image`

保留 `answer` 作为文本兜底字段，兼容旧逻辑、历史存储和复制操作。

### 3. 聊天历史完整保存图片响应结构

`/chat/ask` 的 `response_payload` 现在会持久化 `content_parts`。这意味着：

1. 刷新页面后仍可重放模型返回的图片消息
2. 编辑/重生成后也可继续沿用结构化响应载荷

### 4. 前端聊天区支持渲染模型输出图片

`app/web/app.js` 和 `app/web/styles.css` 已新增消息富内容渲染：

1. 文本块按原顺序显示
2. 图片块以内嵌图片显示
3. 点击图片可新标签打开原图
4. 旧文本消息继续按原样显示
5. 历史消息会根据 `response_payload.content_parts` 重建图片内容

这次改造不会影响原有纯文本消息链路。

### 5. 图片输出与文本输出可混合

如果模型返回：

1. 一段说明文字
2. 一张图片
3. 再补一段说明

前端会按顺序渲染，而不是把所有内容压成单个字符串。

## 兼容性说明

### 1. 向后兼容

旧的纯文本回答无需迁移，仍按 `answer` / `response_text` 正常显示。

### 2. 结构化图片输出的适配边界

本次支持的是：

1. 当前 `chat/completions` 路径下返回的图片块
2. OpenAI-compatible 响应里常见的图片 URL / data URL 结构

如果某个供应商要求：

1. 独立图片生成 endpoint
2. 非标准返回体
3. 额外的 vendor-specific 参数

则还需要针对该供应商补充专门适配。本版本先打通通用展示链路。
