# v0.10.4 - 2026-03-25

## 版本概览

本次修复解决了“模型回复显示不完整、回答在中途被截断”的问题。排查结论如下：

1. 前端主消息区域没有对助手正文做字数截断。
2. 真正的问题在后端，LLM 请求默认 `max_tokens=512`，并且当模型返回 `finish_reason=length` 时，系统直接把半截答案当成最终结果返回。

## 问题根因

### 1. 前端不是截断点

`app/web/app.js` 中主消息渲染逻辑使用的是完整 `textContent` 写入，未对助手最终回答做长度裁剪。

前端里确实存在 `truncate()`，但它只用于“引用来源 snippet”展示，不用于助手正文。

### 2. 后端存在硬限制且缺少续写逻辑

后端原有行为：

1. 每次请求都带 `max_tokens`。
2. 默认值来自配置 `LLM_MAX_TOKENS`，此前默认是 `512`。
3. 当模型因为长度上限停止生成时，返回体里的 `finish_reason=length` 没有被处理。
4. 服务层直接把这一段不完整文本返回给前端，前端只能展示“被截断后的结果”。

## 本次修复

### 1. 增加自动续写机制

在 `app/services/llm_service.py` 中新增了分段补全逻辑：

1. 首次请求拿到回答片段后，检查 `finish_reason`。
2. 如果是 `length`，说明模型不是正常结束，而是被 token 上限截断。
3. 系统自动把当前片段作为上一轮 assistant 内容补回消息列表。
4. 再追加一条明确的续写提示：
   - `Continue exactly from where you stopped. Do not repeat prior text. Finish the current answer.`
5. 继续向模型发起下一段请求。
6. 最多允许 4 段补全，避免异常情况下无限续写。

这样即使运行环境里的 `.env` 仍然保留较小的 `LLM_MAX_TOKENS`，系统也不会再把半截答案直接返回给用户。

### 2. 修正分段拼接细节

在实现自动续写后，又补了一个细节修复：

1. 解析每段回答时，不再提前裁掉段尾空格。
2. 最终只在整段完成后统一 `strip()`。

否则会出现：

- 第一段以 `"Part one. "` 结束
- 第二段以 `"Part two."` 开始
- 拼接后错误变成 `"Part one.Part two."`

现在这类句间空格会被正确保留。

### 3. 提高默认输出上限

为了减少不必要的续写次数：

1. `app/core/config.py` 中默认 `llm_max_tokens` 从 `512` 提高到 `2048`
2. `.env.example` 同步更新为 `LLM_MAX_TOKENS=2048`

说明：

1. 这是默认值提升，不会强行修改用户本地 `.env`
2. 如果当前运行环境里的 `.env` 仍是 `512`，系统仍可依赖“自动续写”拿到完整答案
3. 如果希望减少额外续写调用，可以把本地 `.env` 的 `LLM_MAX_TOKENS` 也改成更高值后重启服务

### 4. 同步版本号

`app/core/config.py` 中应用版本更新为 `0.10.4`，保证运行中的版本元信息与 changelog 一致。

## 未落地提案

本版本修复后，系统已经能避免“半截答案直接返回”的问题。但针对“是否应该彻底取消输出字数上限”，本轮结论是不建议直接取消，而建议升级为企业级的“高上限 + 动态预算 + 截断治理”方案。

提案核心如下：

1. 不采用“无限输出”策略
2. 默认输出上限提升到更合理的范围，例如 `2048` 或 `4096`
3. 后端根据模型上下文窗口、输入 token、系统安全余量动态计算可输出预算
4. 保留 `finish_reason` 检测，不再依赖多轮自动续写作为主路径
5. 自动续写降级为兜底机制，最多执行 1 次
6. 前端补充“继续生成”按钮和“已达到本次输出上限”提示
7. 记录输出预算、实际输出、finish reason、续写次数等可观测性指标

该提案暂不在 `v0.10.4` 落地，仅进入设计阶段。详细方案见：

- `docs/step-11-llm-output-governance-design.md`
