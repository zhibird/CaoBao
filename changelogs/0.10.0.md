# v0.10.0 - 2026-03-23

## 版本概览

本版本实现了附件上传能力升级，目标是让会话附件支持 `txt/md/pdf/png/jpg/jpeg/webp`，并将“解析、切块、索引”从前端下沉到后端统一编排。

核心结果：

1. 新增真实文件上传接口（multipart），支持 PDF 与图片上传。
2. 新增原文件落盘与元数据持久化（MIME、大小、哈希、存储路径、失败信息等）。
3. 新增后端异步处理流水线：`uploaded -> parsing -> chunking -> indexing -> ready/failed`。
4. 前端改为“上传 + 轮询状态”，不再直接调用 chunk/index。
5. 检索与聊天来源补充页码定位信息（`page_no`、`locator_label`）。

## 文件上传逻辑（严格说明）

### 1) 前端发起上传

前端不再读取 `file.text()`，改为 `FormData` 直传原始二进制文件：

1. 入口：`+` 菜单选择“上传文件”。
2. 请求：`POST /api/v1/documents/upload`。
3. 表单字段：
   - `team_id`
   - `user_id`
   - `conversation_id`
   - `auto_index=true`
   - `embedding_model`
   - `file`（二进制）

### 2) 后端上传校验

上传接口接收后，按以下顺序严格校验：

1. 文件是否为空（`EMPTY_FILE`）。
2. 文件大小是否超过限制（默认 20MB，`FILE_TOO_LARGE`）。
3. 扩展名是否在允许列表（`txt/md/pdf/png/jpg/jpeg/webp`，否则 `UNSUPPORTED_FILE_TYPE`）。
4. magic bytes 与扩展名是否匹配：
   - PDF: `%PDF-`
   - PNG: `89 50 4E 47 0D 0A 1A 0A`
   - JPEG: `FF D8 FF`
   - WEBP: `RIFF....WEBP`
5. 浏览器声明 MIME 与后端嗅探 MIME 是否冲突（冲突则拒绝）。
6. 文本文件必须 UTF-8（否则 `UNSUPPORTED_TEXT_ENCODING`）。

### 3) 落盘与入库

校验通过后执行：

1. 生成 `document_id`。
2. 将原文件落盘到：`UPLOAD_ROOT_DIR/<team_id>/<document_id>/original/<sanitized_name>`。
3. 写入 `documents` 元数据：
   - `content_type`、`mime_type`、`size_bytes`、`sha256`
   - `storage_key`
   - `status=uploaded`
   - `content` 初始为空（等待解析）
   - `created_at`、`updated_at`
4. 立即返回 `201`（文档状态为 `uploaded`）。

## 后端处理流水线逻辑（严格说明）

上传或文本导入后（`auto_index=true`）进入后台任务 `_process_document_pipeline_task`。

### 阶段 A：解析（`parsing`）

1. 状态更新为 `parsing`。
2. 按类型解析：
   - `txt/md`：UTF-8 解码后写入 `documents.content`。
   - `pdf`：提取文本层；若未提取到文本，则写入可检索占位文本（保证链路可继续）。
   - `png/jpg/jpeg/webp`：
     - 若 OCR 环境可用（pytesseract + tesseract）则尝试 OCR；
     - 若 OCR 不可用或失败，降级为占位文本 `Image attachment: <source_name>`。
3. 写入解析元信息 `meta_json`、`page_count`。
4. 解析失败则写入：
   - `status=failed`
   - `failure_stage=parse`
   - `error_code/error_message`

### 阶段 B：切块（`chunking`）

1. `ChunkService` 先清除该文档旧 chunks 和 embeddings（重建保障）。
2. 使用现有切块策略（max_chars/overlap）对解析文本切块。
3. 若解析有分页信息，则将页码映射到 chunk：
   - `document_chunks.page_no`
   - `document_chunks.locator_label`（如 `Page 3`）
   - `document_chunks.block_type/meta_json`
4. 成功后文档状态回到 `uploaded`（等待索引）。
5. 失败则：
   - `status=failed`
   - `failure_stage=chunk`
   - `error_code/error_message`

### 阶段 C：索引（`indexing`）

1. `RetrievalService.index_chunks(..., rebuild=True)` 对目标文档重新建索引。
2. 批量生成向量并写入 `chunk_embeddings`。
3. 成功后状态置为 `ready`。
4. 失败则：
   - `status=failed`
   - `failure_stage=index`
   - `error_code/error_message`

## 文本导入逻辑变更

`POST /api/v1/documents/import` 保留，用于粘贴文本。

变更点：

1. 返回状态从旧的 `pending` 改为 `uploaded`。
2. 支持 `auto_index`（默认 `false`，前端传 `true`）。
3. 支持传入 `user_id` 与 `embedding_model`，用于后端索引编排。

---

## 前端交互逻辑变更

1. 上传文件：调用 `/documents/upload`，不再 `file.text()`。
2. 粘贴导入：调用 `/documents/import` 后不再手动调用 `/documents/{id}/chunk` 与 `/retrieval/index`。
3. 轮询策略：
   - 每 2 秒刷新会话文档列表
   - 直到目标文档进入 `ready/failed` 终态
4. 状态展示扩展为：
   - `uploaded/parsing/chunking/indexing/ready/failed/deleted`
5. 来源展示支持 `page_no/locator_label`。

## 数据模型与迁移

### documents 新增列

1. `mime_type`
2. `size_bytes`
3. `sha256`
4. `storage_key`
5. `preview_key`
6. `page_count`
7. `failure_stage`
8. `error_code`
9. `error_message`
10. `meta_json`
11. `updated_at`

### document_chunks 新增列

1. `page_no`
2. `locator_label`
3. `block_type`
4. `meta_json`

### 迁移策略

1. 轻量补列（best effort）。
2. 回填历史文本导入数据：
   - `mime_type`
   - `size_bytes`
   - `sha256`
   - `storage_key=inline://legacy/<document_id>`
   - `updated_at`
3. 补充索引以支持会话筛选与状态查询。
