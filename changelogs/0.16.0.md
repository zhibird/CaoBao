# v0.16.0 - 2026-04-13

## 前情与承接

`v0.16.0` 承接了 `v0.14.4` 的工作台入口收敛，以及 `v0.15.0` 的数据库主路径规范化，但项目在用户登录鉴权这一层仍然存在明显缺口：

1. 前端虽然已经不再暴露早期的开发者式入口，但真实身份仍然依赖本地 `team_id / user_id` 切换，缺少标准登录态来源。
2. 后端大量业务接口已经按团队与用户做隔离，却没有一条完整的“注册 -> 登录 -> 当前会话 -> 续期 -> 登出 -> 失效”的标准闭环。
3. 历史本地 `SQLite` 数据库在切入新鉴权逻辑后，还存在 schema 兼容风险；前端对结构化校验错误的提示也不够可读。

本次版本的目标，就是把“工作台身份认领”正式升级为“标准登录鉴权闭环”，并把旧本地库与前端交互层的兼容问题一起补齐。

## 版本概览

本版本主要完成：

1. 新增标准登录鉴权闭环，前端与后端统一改为基于会话 cookie 的真实身份。
2. 业务接口正式从当前登录用户推导 `team_id / user_id`，不再信任前端随意传入的身份参数。
3. 补齐旧 `SQLite` 本地库兼容升级、鉴权报错可读性和静态资源缓存刷新链路。

## 主要更新

1. 新增 `register / login / me / refresh / logout / change-password` 认证接口与完整会话管理。
2. 前端工作台入口改为标准“登录 / 注册”认证弹层，并在应用启动时优先探测当前会话。
3. Access Token + Refresh Token 双 cookie 会话链路落地，支持短期访问、自动续期、主动登出与密码变更失效。
4. 业务路由统一接入 `require_current_active_user`，从服务端登录态兜底团队与用户身份。
5. 补齐旧 `SQLite` `users` 表 auth 字段兼容补列，避免本地历史库在注册链路上直接 `500`。
6. 前端 auth 表单新增本地密码规则校验，并把 FastAPI `422` 结构化错误格式化为可读提示，不再显示 `[object Object]`。

## 新增内容

1. 新增认证模型与 schema：`AuthRefreshSession`、`RegisterRequest`、`LoginRequest`、`AuthSessionResponse`。
2. 新增认证服务层：密码哈希、JWT access token 生成、refresh token 轮换、登出撤销、密码变更后批量失效。
3. 新增认证接口：`POST /api/v1/auth/register`、`POST /api/v1/auth/login`、`GET /api/v1/auth/me`、`POST /api/v1/auth/refresh`、`POST /api/v1/auth/logout`、`POST /api/v1/auth/change-password`。
4. 新增前端登录注册 UI、登录态启动引导、失效后重新登录、切换账号与退出登录入口。
5. 新增 auth 相关测试与前端静态回归，覆盖 cookie 写入、会话轮换、登出失效、认证 UI 与错误提示。

## 处理链路 / 流程说明

1. 应用启动后先请求 `GET /api/v1/auth/me`，若存在有效会话则直接恢复当前用户与工作台。
2. 若未登录，则打开标准认证弹层，用户可执行注册或登录，而不再通过输入 ID 直接认领工作区。
3. 登录或注册成功后，后端写入 `HttpOnly` access / refresh cookies，前端再统一加载模型、会话、资料与收藏数据。
4. 受保护接口若遇到 access token 过期，会尝试一次 `POST /api/v1/auth/refresh`，成功后重放原请求；失败则回到未登录状态。
5. 用户主动登出或修改密码后，refresh session 会失效，旧会话无法继续恢复。

## 问题现象

1. 旧版本前端仍然沿用 `team_id / user_id` 作为真实身份来源，缺少标准登录态闭环。
2. 历史本地 `SQLite` 库在接入 auth 后，`users` 表缺少 `password_hash / is_active / password_updated_at`，导致注册链路直接报错。
3. 前端遇到 FastAPI `422` 校验错误时，会把结构化错误对象直接渲染成 `[object Object],[object Object]`，用户无法理解真实原因。
4. 静态资源版本串未更新时，浏览器容易继续复用旧 `app.js / styles.css` 缓存，导致界面看起来“和以前没有区别”。

## 根因分析

1. 项目原有身份链路是“工作台切换模型”，并不是“标准认证模型”，因此没有统一的 session 来源与失效规则。
2. 虽然 Alembic 已引入 auth 迁移，但 `dev + sqlite` 仍会走 legacy bootstrap 路径，老 `users` 表不会自动补齐 auth 字段。
3. 前端错误提取函数只按字符串处理 `detail`，没有格式化 FastAPI 常见的数组型校验错误。
4. 静态资源 query build 未推进时，浏览器无法被稳定地强制刷新到最新前端脚本。

## 修复内容

1. 后端补齐标准认证能力，包括密码哈希、access token、refresh token、refresh session 轮换、登出撤销与密码变更失效。
2. 业务路由统一改为从当前登录用户推导 `team_id / user_id`，接口不再把前端传入身份当成真实来源。
3. 前端认证入口改为登录/注册弹层，启动时以 `/auth/me` 为唯一真实登录态来源。
4. 前端新增 `401 -> refresh -> 重放一次请求` 的统一续期逻辑，并在彻底失效时回到登录态。
5. legacy `SQLite` 初始化逻辑新增 `users` 表 auth 字段补齐，兼容现有本地历史数据库。
6. auth 表单新增密码长度与确认密码一致性校验，FastAPI `422` 错误改为人类可读提示。
7. 前端静态资源 build 串升级，降低旧缓存持续命中新脚本不可见的风险。

## 交互与 UI 调整

1. 旧的“进入 / 切换工作台”入口被替换为标准“登录 / 注册”认证入口。
2. 设置面板新增“切换账号 / 退出登录”，而不再是直接切换工作区身份。
3. 未登录访问受保护能力时，会统一回到认证弹层，而不是继续沿用本地伪身份。
4. 登录与注册表单会直接提示“密码至少需要 8 个字符”“两次输入的密码不一致”等明确错误。

## API / 数据模型 / 配置变更

1. 新增 auth 路由与 `AuthService`，引入基于 cookie 的认证会话管理。
2. `users` 表新增 `password_hash`、`is_active`、`password_updated_at` 字段。
3. 新增 `auth_refresh_sessions` 表，用于 refresh token 摘要存储、轮换与撤销。
4. `Settings` 新增 auth 相关配置：JWT secret、cookie 名称、cookie secure / samesite、access / refresh TTL。
5. 前端 `requestJson / apiRequest` 统一使用 `credentials: "same-origin"` 携带 cookie。

## 兼容性说明

1. 当前版本保留部分请求中的 `team_id / user_id` 兼容字段，但后端将以当前登录用户为准，不再信任客户端身份来源。
2. `X-Dev-Admin-Token` 相关开发者入口未在本次版本中移除，避免与主工作台用户鉴权链路混淆。
3. 历史本地 `SQLite` 数据不会被清空，版本已增加对旧 `users` 表 auth 字段的兼容补齐。
4. 浏览器若曾缓存旧前端资源，需在首次升级后重新加载一次页面，以拿到新的静态资源 build。
