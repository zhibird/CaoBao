# v0.14.0 - 2026-04-01

## 前情与承接

`v0.13.0` 已经把 CaiBao 从“单轮聊天”推进到“同会话连续对话”：最近若干轮 `ask/echo` 历史会进入模型上下文，消息编辑边界也被收紧到了“仅最后一条消息”。这一步解决的是“同一个会话里，模型能不能延续上下文”的问题。  
但产品在更高一层的“长期记忆与知识沉淀”上仍然存在明显空缺：

1. 附件和检索仍主要停留在会话级，缺少“同一个项目内跨会话复用”的正式边界。
2. 用户虽然能得到有价值回答，但系统没有“收藏、升格、沉淀、复用”的闭环。
3. 一些长期约束、偏好、项目事实，只能靠用户反复重复输入，无法形成结构化记忆。
4. 数据库演进此前主要依赖 `create_all + ALTER TABLE` 的轻量方式，服务能跑，但不适合作为后续复杂版本的正式部署基线。

`v0.14.0` 承接的就是这几条主线：  
一方面，把系统从“会话内记忆”升级为“带项目边界的长期知识与记忆工作区”；另一方面，把部署链路从“代码能启动”升级为“schema 可版本化、上线可重复、容器可自举迁移”。

## 版本概览

本版本是 CaiBao 在信息资产能力上的一次大升级，核心不是单点新增一个接口，而是搭起一整套“项目空间 -> 知识库 -> 记忆卡 -> 收藏 -> 结论 -> 检索复用”的分层体系，并同步补齐对应的迁移与部署基础设施。

本版本主要完成：

1. 新增项目空间（`project_spaces`）与空间边界，让跨会话复用不再是“全历史互通”，而是“同空间内可控复用”。
2. 新增记忆卡（`memory_cards`）与向量索引（`memory_card_embeddings`），让长期偏好、项目约束、稳定事实可以结构化管理并在问答前注入。
3. 新增回答收藏（`answer_favorites`）与历史结论（`conclusions`）沉淀链路，支持从收藏转记忆卡或结论，并把有效结论同步回可检索知识文档。
4. 正式接通 `include_conclusions` 到聊天链路，使结论不再只是后台资产，而能真实参与问答上下文。
5. 引入 Alembic-first 的生产迁移基线，补齐容器启动前自动迁移、schema gate 与历史库 stamp 对齐策略。

## 主要更新

1. 聊天系统正式具备“跨会话但有项目边界”的长期上下文能力，问答不再只依赖当前会话和临时附件。
2. 数据资产从单一 `documents` 语义拆分为多层：知识库、记忆卡、收藏、结论各自承担不同职责，避免所有内容混入同一个检索池。
3. 收藏与结论不只是 CRUD，而是形成了“回答快照 -> 收藏 -> 升格为记忆/结论 -> 进入长期复用链路”的闭环。
4. 生产部署由“启动时顺手建表”切换为“先迁移、后启动”，使后续版本迭代具备更可靠的发布基础。

## 设计与架构思路

### 为什么这次不是继续把能力堆在 `documents` 上

从数据形态看，附件、记忆、收藏、结论并不是同一种东西：

1. 附件 / 知识库文档是原始资料，适合切块、建索引、做 RAG。
2. 记忆卡是短小、结构化、可编辑、可过期的长期记忆单元，适合在回答前快速注入。
3. 收藏是回答快照，本质是书签，不应该默认参与自动检索。
4. 结论是对历史对话的加工与确认结果，价值高于收藏，但又不同于原始知识材料。

如果把这四类东西全部塞进 `documents`，虽然表面上“统一了”，但会带来几个问题：

1. 生命周期混乱：附件、结论、记忆的失效方式完全不同。
2. 用户心智混乱：难以解释“为什么这个东西会被召回”。
3. 检索噪声增大：低质量回答或草稿结论会污染知识库召回结果。
4. 编辑体验变差：结构化记忆卡不适合被当作文档去处理。

所以本版本坚持分层设计：  
`documents` 继续负责长期可检索资料；`memory_cards` 负责结构化记忆；`answer_favorites` 负责轻量收藏；`conclusions` 负责高价值历史结论沉淀。

### 为什么引入项目空间

此前系统主要依赖 `team_id / user_id / conversation_id` 组织数据，这对“单会话聊天”足够，但对“长期工作区”不够。  
`project_spaces` 的作用不是简单分组，而是定义一个正式的长期上下文边界：

1. 一个空间下可以有多个会话。
2. 一个空间下可以沉淀知识文档、记忆卡、收藏和结论。
3. 默认跨会话调用只发生在同一个空间内，而不是所有历史会话全开。

这使系统终于具备了“围绕一个项目持续工作”的能力，而不是每次都从新会话重新开始。

## 核心架构变化

### 1. 数据模型层

本版本新增或强化了以下核心实体：

1. `project_spaces`
   - 负责项目空间边界、默认空间、用户可访问范围。
   - `conversations / documents / chat_history / memories / favorites / conclusions` 都开始与 `space_id` 发生关联。
2. `memory_cards`
   - 保存偏好、约束、目标、事实等长期记忆。
   - 每张卡可独立启用、停用、过期、更新。
3. `memory_card_embeddings`
   - 为记忆卡提供独立向量索引，不与文档 chunk 索引混用。
4. `answer_favorites`
   - 保存回答快照、原问题、原回答、来源快照、备注和标签。
5. `conclusions`
   - 保存历史结论的主题、正文、可信度、状态、证据和同步文档 ID。

同时，既有模型也被扩展：

1. `conversations`、`documents`、`chat_history` 挂接 `space_id`。
2. `documents` 增强了 `visibility / asset_kind / retrieval_enabled / origin_document_id` 等语义，用于承载知识库、结论文档等长期资产。

### 2. 服务层分工

这次改动没有把所有逻辑继续塞在聊天服务里，而是引入了更明确的服务职责划分：

1. `SpaceService`
   - 负责默认空间创建、空间访问控制、空间级 CRUD。
2. `MemoryService`
   - 负责记忆卡 CRUD、embedding 写入、记忆检索。
3. `FavoriteService`
   - 负责回答收藏及“转记忆 / 转结论”。
4. `ConclusionService`
   - 负责结论 CRUD、确认、归档，以及同步为 `conclusion_note` 文档。
5. `DocumentService`
   - 继续负责文档上传、解析、切块、索引、发布到知识库等基础能力。
6. `RagChatService`
   - 继续作为聊天入口，但上下文已不再只来自文档，而是组合当前会话、记忆卡、知识库与结论文档。

这个分层的好处是，资产沉淀和聊天编排不再互相强耦合，后续要继续做自动化建议、治理、版本替换时也更容易扩展。

## 处理链路 / 流程说明

### 1. 会话附件发布到知识库

本次不是简单把原附件“改个状态”，而是明确走“复制为长期资产”的策略：

1. 会话中上传的附件仍保持会话语义。
2. 当用户执行“发布到知识库”时，系统复制一条新的 `documents` 记录。
3. 新文档的 `visibility=space`，`asset_kind=knowledge_doc`。
4. 原始附件与新知识文档通过 `origin_document_id` 建立来源关联。
5. 新文档继续复用现有切块、向量索引、预览能力。

这样设计的原因是：删除会话附件时，不应该误伤已经发布到知识库的长期资产。

### 2. 记忆卡注入链路

聊天发起时，系统现在会先做空间边界解析，再决定是否检索记忆卡：

1. 基于 `team_id + user_id + conversation_id + space_id` 解析有效空间。
2. 若 `include_memory=true`，则对当前问题生成 embedding。
3. 在当前 `space_id` 下搜索可用、未过期、处于激活状态的记忆卡。
4. 将命中的少量高价值记忆以结构化文本形式注入聊天上下文。

这里的重点不是“把所有记忆全丢给模型”，而是保持小而精，优先服务稳定偏好与项目约束。

### 3. 收藏与结论沉淀链路

本版本让回答拥有了正式的“升格路径”：

1. 用户先把一条回答保存为 `favorite`。
2. 收藏会保存原问题、原回答，以及当时的来源快照。
3. 收藏可以一键转为 `memory card`，适合长期偏好、术语约定、项目事实。
4. 收藏也可以一键转为 `conclusion`，适合经过整理后的决策与方案沉淀。

这里“收藏”和“结论”的分层非常关键：

1. 收藏默认不直接进检索。
2. 结论必须经过确认后，才有资格被同步为长期知识资产。

### 4. 结论进入聊天链路

`include_conclusions` 在设计稿里早就存在，但此前只是 schema 字段，没有真实接进问答。  
本版本正式打通了这条链路，而且采用的是“复用文档检索链路”的策略，而不是重做一套独立结论向量栈：

1. 结论在 `confirmed / effective` 状态下，会同步为 `documents` 中的 `conclusion_note`。
2. 同步后的结论文档会自动走 chunk + index 流程，直接变成可检索资料。
3. `chat/ask` 在 `include_conclusions=true` 时，会把这些 `conclusion_note` 纳入检索作用域。
4. 当 `include_conclusions=false` 时，行为与旧版保持一致，不会把结论带入上下文。

这个设计最大的优点是：  
结论最终仍然走已有 RAG 基础设施，减少了系统内部“第二套搜索路径”的维护成本。

   - `GET /api/v1/spaces`
   - `PATCH /api/v1/spaces/{space_id}`
   - `DELETE /api/v1/spaces/{space_id}`
2. 知识库：

   - `POST /api/v1/library/documents/publish-from-conversation`
3. 记忆卡：
   - `POST /api/v1/memory/cards`
   - `GET /api/v1/memory/cards`
   - `PATCH /api/v1/memory/cards/{memory_id}`
   - `DELETE /api/v1/memory/cards/{memory_id}`
4. 收藏：
   - `POST /api/v1/favorites/answers`
   - `GET /api/v1/favorites/answers`
   - `PATCH /api/v1/favorites/answers/{favorite_id}`
   - `DELETE /api/v1/favorites/answers/{favorite_id}`
   - `POST /api/v1/favorites/answers/{favorite_id}/promote-to-memory`
   - `POST /api/v1/favorites/answers/{favorite_id}/promote-to-conclusion`
5. 结论：
   - `POST /api/v1/conclusions`
   - `GET /api/v1/conclusions`
   - `PATCH /api/v1/conclusions/{conclusion_id}`
   - `POST /api/v1/conclusions/{conclusion_id}/confirm`
   - `POST /api/v1/conclusions/{conclusion_id}/archive`

### 聊天接口变化

`POST /api/v1/chat/ask` 增强了工作区上下文控制能力：

1. `space_id`
2. `include_memory`
3. `include_conclusions`
4. `include_library`

本版本最重要的变化是：  
`include_conclusions` 从“声明存在但未生效”变为“真实控制结论召回”。

### 配置与部署变化

本版本增加了生产部署相关约束：

1. 生产环境改为 Alembic-first。
2. 容器启动前会先执行 `alembic upgrade head`。
3. 若数据库 revision 不是 `head`，应用会拒绝启动。
4. 新增 `DB_LEGACY_INIT_ENABLED`，用于显式控制是否启用旧的 `create_all + ALTER` 兜底路径。

## 部署与迁移架构调整

这是 0.14.0 中非常重要、但容易被功能点掩盖的一部分。

### 为什么要调整迁移策略

随着 `memory_cards`、`favorites`、`conclusions`、`project_spaces` 等实体出现，原先的轻量迁移方式已经接近极限：

1. `create_all` 更适合全新空库，不适合严格版本演进。
2. 手写 `ALTER TABLE` 只能解决少量字段补丁，无法优雅支撑复杂 schema 演进。
3. 容器内如果没有正式 migration 流程，部署顺序会依赖人工操作，风险很高。

### 本版本的处理方式

本版本把部署基线正式转为 Alembic-first：

1. 新增 Alembic bootstrap revision，补齐核心表基线。
2. 调整已有 revision，使空库、历史库和半升级库都能更稳定地对齐到 `head`。
3. 容器镜像纳入 `alembic/` 与 `alembic.ini`。
4. Docker entrypoint 会先执行迁移，再启动 uvicorn。
5. 应用在 `APP_ENV=prod` 且 schema 未对齐时会直接拒绝启动。

### 历史库兼容策略

为了避免旧库直接升级时因“表已存在”报错，本版本文档明确给出 `stamp + upgrade` 对齐方案：

1. 旧库先 `alembic stamp head` 或先 stamp 到基线。
2. 再执行 `alembic upgrade head` 做一致性校验。

这使系统第一次真正有了“可重复部署”的版本化路径，而不是每次靠环境状态碰运气。

## 权限与边界治理

除了新增能力，本版本还顺手补了几个非常关键的边界问题：

1. `chat/ask` 在解析 `conversation_id / space_id` 时，现在会校验会话确实属于当前用户。
2. 结论与知识库的召回严格受 `space_id` 约束，不会因为“同 team”就默认可跨空间读取。
3. 当当前作用域内没有可检索文档时，RAG 不会错误退化为“全 team 检索”。

这类修正对用户不可见，但对数据安全和结果可信度很重要，尤其是在“长期资产”开始增多之后。

## 兼容性说明

1. 当前生产数据库基线仍为 SQLite，本版本没有切换到 PostgreSQL。
2. 结论检索复用 `documents(conclusion_note)`，没有新增独立结论向量检索栈。
3. 旧的本地开发方式仍可通过 `DB_LEGACY_INIT_ENABLED` 显式保留，但生产建议关闭。
4. 前端完整资产面板、自动沉淀建议、定时治理和版本替换不在本版本交付范围内。

## 后续建议

1. 在 `v0.14.x` 后续小版本中，可以继续补 `memory/cards/suggest-from-message`，把“从消息生成记忆草稿”这段链路闭上。
2. 如果后续要强化可解释性，可以为 `ChatSource` 补充 `source_type / source_id / reason`，把文档、记忆、结论三类来源明确区分开。
3. 当 Phase 4 进入实施时，建议重点做：
   - 自动化沉淀建议
   - 过期记忆治理
   - 重复结论检测
   - 结论版本替换与归档策略
