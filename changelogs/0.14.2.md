# v0.14.2 - 2026-04-05

## 前情与承接

`v0.14.1` 已经把 CaiBao 的首页定位、默认直聊语义、资料按需增强模式和历史编辑链路统一到了同一条主线上。  
但产品经理在此基础上又指出了一个更关键的 P1：

1. 后端与设计层已经具备 `spaces / library / memory / favorites / conclusions` 这些长期资产能力。
2. 前端回答卡片主操作区却仍然只有“复制 / 重新生成”，用户无法在回答出现的当下完成沉淀动作。
3. 这会导致产品虽然已经具备“会成长”的后端能力，但用户体验仍停留在一次性问答，而不是可持续积累的工作台。

`v0.14.2` 承接的就是这一段能力落差：  
不是继续扩展抽象概念，而是把“回答卡片 -> 收藏 / 记忆 / 结论 / 资料库”这条最有复利价值的链路真正挂到主流程上，并补齐追溯、状态和回归验证。

## 版本概览

本版本主要完成：

1. 回答卡片主操作区正式接入长期资产闭环，用户可直接从回答出发完成沉淀。
2. 前端补齐收藏、记忆、结论、资料库的状态回显与来源追溯，不再是“点完就失忆”的一次性动作。
3. 后端补齐 `source_message_id / meta` 相关语义，确保沉淀资产可追溯回原始消息，并通过全量回归验证。

## 主要更新

1. 回答卡片新增 `收藏 / 记住这条 / 沉淀为结论 / 发布到资料库 / 复制 / 重新生成` 主操作。
2. 前端会按当前 `space` 拉取 favorites / memory / conclusions / library 状态，按钮可显示 `已收藏 / 已记住 / 已成结论 / 已入资料库`。
3. 资料库沉淀改为支持直接从回答生成知识文档，并保留来源消息元数据。
4. 记忆卡与结论创建链路都新增来源消息校验，避免写入无效或跨用户来源。
5. 静态资源版本号已更新，避免浏览器继续命中旧缓存而看不到本次前端变更。

## 问题现象

1. 用户在回答卡片上只能“复制 / 重新生成”，无法顺手把有价值内容沉淀为长期资产。
2. 即使后端已有 favorites / memory / conclusions / library，用户在首条关键回答出现时也没有入口可用。
3. 前端没有状态回显时，用户无法判断一条回答是否已经收藏或转化为长期资产。
4. 浏览器若继续命中旧版 `/web/app.js` 与 `/web/styles.css`，会表现为“代码已改，但页面没有任何变化”。

## 根因分析

1. 资产型能力存在于后端与设计文档中，但没有进入回答卡片的高频主交互。
2. `memory / conclusion / library` 的内部数据语义已经存在，但前端没有把它们收束成“来源清晰、状态可见”的操作链。
3. 文档导入与解析链路此前会覆盖 `meta_json`，导致“来源消息”这类自定义元数据无法稳定保留。
4. 首页静态资源版本号没有随着本次前端改动同步更新，浏览器缓存会放大“像是没更新”的错觉。

## 修复内容

### 1. 回答卡片主操作区接入资产闭环

1. 在 assistant 消息卡片主操作区接入：
   - `收藏`
   - `记住这条`
   - `沉淀为结论`
   - `发布到资料库`
   - `复制`
   - `重新生成`
2. 回答卡片的主操作区改为默认可见，避免这些关键动作只在 hover 时才出现。
3. 窄屏下支持换行，避免在移动端出现六个动作一字排开导致不可用。

### 2. 前端状态回显与并发保护

1. 新增 `messageCaptures` 前端状态，分别按 `message_id` 维护收藏、记忆、结论和资料库映射。
2. 切换会话后会按当前 `space` 拉取已有资产状态，回答卡片上实时显示“已收藏 / 已记住 / 已成结论 / 已入资料库”。
3. 对 capture 状态加载增加会话代际校验，避免用户快速切换会话时旧请求回写错误状态。
4. 回答卡片状态刷新改为保留当前阅读位置，不再因为点击沉淀按钮把长对话强制滚到最底部。

### 3. 后端追溯语义补齐

1. `MemoryCardCreate / MemoryCardResponse` 新增 `source_message_id`。
2. `MemoryService.create()` 新增来源消息校验；从 favorite 提升到 memory 时会自动带上原始 `message_id`。
3. `ConclusionService.create()` 新增来源消息校验，保证 `source_message_id` 必须属于当前 team/user。
4. `DocumentImportRequest` 新增 `meta` 字段，用于从回答沉淀到资料库时写入 `source_message_id` 等元数据。
5. `DocumentService` 新增 `meta_json` 合并逻辑，解析阶段不再覆盖导入时写入的来源元数据。

## 交互与 UI 调整

1. assistant 回答卡片主操作区从“复制 / 重新生成”扩展为“沉淀优先”的主操作区。
2. assistant 卡片操作区默认可见，user 卡片仍保持更轻量的操作密度。
3. 回答卡片动作支持已完成状态高亮，帮助用户建立“这条已经沉淀过”的感知。
4. 静态资源版本号升级为 `0.14.2 / 20260405-1`，便于浏览器刷新后立即命中新前端。

## 兼容性说明

1. 本版没有移除既有 favorites / memory / conclusions / library 后端能力，只是把它们显式挂到了回答卡片主流程上。
2. 对既有文档导入逻辑保持兼容：`meta` 为可选字段，不影响旧调用。
3. 本版不涉及数据库迁移，新增语义建立在现有字段和模型能力之上。
4. 用户若仍看不到新操作，优先检查是否命中了旧静态资源缓存；本版已通过版本号更新降低该问题发生概率。

## 下版本做什么

本版虽然已经把“收藏 / 记住这条 / 沉淀为结论 / 发布到资料库”接入主流程，但产品表达上仍偏重系统视角。  
结合当前评审反馈，下一版建议收束为：

1. 在回答卡片首层只保留 `收藏`。
2. 将 `设为长期记忆 / 沉淀为结论 / 发布到资料库` 下沉到收藏后的整理页或收藏详情页。
3. 用“先收藏，后整理”的统一口径，替代用户在回答出现当下就被迫做资产分类的交互负担。

> 也就是说：  
> `收藏` 保留在主流程；  
> `记忆 / 结论 / 资料库` 保留在系统能力层，但下一版更适合下沉，而不是继续并列暴露在回答卡片首层。
