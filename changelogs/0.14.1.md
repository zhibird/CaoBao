# v0.14.1 - 2026-04-05

## 前情与承接

`v0.14.0-beta` 已经把 CaiBao 的长期资产能力搭了起来：项目空间、知识库、记忆卡、收藏、结论与跨会话上下文都已经进入正式设计和实现轨道。  
但在产品表达层，首页和首屏交互却出现了一次明显“跑偏”：

1. 设计稿要的是“像豆包 / GPT 一样先能直接聊，再按需引入记忆 / 资料库”。
2. 前端首屏却把产品讲成了“资料驱动工作台”，用户会自然理解成“先上传文件才有价值”。
3. 后端实际又已经支持无资料直聊，能力和文案、默认交互并不一致。

`v0.14.1` 承接的就是这个 P1：  
不是继续堆新功能，而是把“产品定位、首页文案、前端模式、后端契约、回归测试”统一收口到同一条主线上，避免系统能力已经具备，但用户第一眼心智被带偏。

## 版本概览

本版本是一次定位收敛与链路校正更新，核心目标不是新增模块，而是把 CaiBao 明确表达为“能直接聊、资料可增强的个人 AI 助手”。

本版本主要完成：

1. 统一一句话定位，首页与设计文档不再混用“个人 AI 助手”和“资料工作台”两套心智。
2. 前端显式引入“直接聊天 / 资料增强”模式切换，默认直聊，资料增强由用户主动开启。
3. 后端补齐 `use_document_scope` 的显式语义，确保前端切到“直接聊天”后不会被隐式文档作用域拉回 RAG。
4. 为“默认直聊”“显式 false 仍走 chat”“历史编辑保留模式”补上回归测试。

## 主要更新

1. 首屏定位统一为“能直接聊、资料可增强的个人 AI 助手”，删除会暗示“必须先上传资料”的表达。
2. 聊天输入区新增模式切换开关，明确告诉用户：默认先聊，需要依据时再显式开启资料增强。
3. 会话内文件虽然仍可上传和检索，但默认不自动参与回答，只有在“资料增强”模式下才进入请求载荷。
4. 历史消息编辑时会保留原始的检索模式与相关上下文字段，避免“改一条消息，整条链路换了语义”。

## 问题现象

1. 首页标题、描述和场景卡更像“资料问答工作台”，会让新用户误判 CaiBao 的起点门槛。
2. 当会话里已有 ready 文件时，旧前端表达容易让人误以为回答默认就会自动检索资料。
3. 后端在兼容旧调用时，如果只看 `document_id / selected_document_ids`，会让“显式关闭资料作用域”的请求重新落回 RAG。
4. 历史编辑链路如果不保留原始上下文控制字段，重新生成后的回答模式可能与首发请求不一致。

## 根因分析

1. 产品定位没有在设计文档、静态首页文案、交互提示和真实请求语义之间形成单一口径。
2. 前端此前更强调“资料在场”，没有明确告诉用户“直聊是默认能力，资料只是增强能力”。
3. 后端新增 `use_document_scope` 后，仍需要兼容老调用方式；如果没有区分“字段未传”和“显式传 false”，就会破坏显式关闭语义。
4. 聊天链路已经从单一请求演进为“消息历史 + 会话文件 + 可选增强”的组合链路，任何一个环节的默认值漂移，都会造成体验和能力不一致。

## 修复内容

### 1. 首页与设计定位统一

1. 在个人版架构设计文档中新增一句话定位：`CaiBao 是一个能直接聊、资料可增强的个人 AI 助手。`
2. 将产品原则补充为：首屏和默认引导必须优先传达“先聊天，再按需引入记忆 / 资料库”。
3. 首页品牌区、顶部标题、Hero 标题、副标题、场景卡和输入框提示全部改为“先聊后增强”的表达。

### 2. 前端模式显式化

1. 新增 `chatMode` 前端状态，分为 `chat` 与 `docs` 两种模式。
2. 输入区增加“直接聊天 / 资料增强”切换按钮。
3. 默认模式为 `chat`，创建新会话、切换会话、登录、会话文件清空或无 ready 文件时都会自动回到 `chat`。
4. 仅在 `docs` 模式下，请求体才会带上 `use_document_scope=true` 和 `selected_document_ids`。
5. “资料增强问答”场景卡在没有 ready 文件时会静默回退到直聊，不再制造“点了就报错”的错误感。

### 3. 后端请求语义收口

1. `ChatAskRequest.use_document_scope` 改为可选字段，用于区分“未传”和“显式传 false”。
2. `RagChatService.ask()` 现在遵循：
   - 显式 `true`：进入资料作用域；
   - 显式 `false`：严格走直聊；
   - 字段未传：仅对显式带文档 id 的旧调用做兼容推断。
3. 历史编辑请求新增并保留 `use_document_scope / include_memory / include_conclusions / include_library`，重新生成时不会丢失原模式。

### 4. 回归测试补齐

新增 3 组针对本次 P1 的关键测试：

1. 即使会话里已有 ready 文件，只要 `use_document_scope=false`，仍必须返回 `mode=chat`。
2. 编辑历史消息时，若原消息是直聊模式，重新生成后仍保持直聊模式。
3. 当请求未传 `use_document_scope` 时，默认走 chat 模式，不再因为“会话里碰巧有文件”就自动变成 RAG。

## 交互与 UI 调整

1. 首页主标题改为“能直接聊、资料可增强的个人 AI 助手”。
2. Hero 改为“登录后就能直接聊，资料会在需要时增强回答”。
3. 场景卡从“总结资料 / 知识问答”调整为“直接聊聊 / 写方案 / 资料增强问答”。
4. 输入区新增模式提示文案，明确说明资料增强是显式开启的，而不是默认行为。
5. 附件条提示也随模式变化：
   - `chat` 模式下说明“文件暂不参与回答”；
   - `docs` 模式下说明“默认检索全部 ready 文件或仅检索选中文件”。
6. 当没有 ready 文件时，“资料增强”按钮会禁用，但用户仍可继续直接聊天。

## API / 数据模型 / 配置变更

1. `app/schemas/chat.py`
   - `ChatAskRequest.use_document_scope` 改为 `bool | None`
   - `ChatHistoryEditRequest` 新增：

     - `include_memory`
     - `include_conclusions`
     - `include_library`
2. `app/services/rag_chat_service.py`
   - 新增“显式优先、未传再兼容”的资料作用域判定逻辑。
3. `app/api/routes/chat.py`
   - 历史编辑链路在重建 `ChatAskRequest` 时保留原始上下文控制字段。

## 兼容性说明

1. 对旧调用保持有限兼容：如果客户端未传 `use_document_scope`，但显式传了 `document_id` 或 `selected_document_ids`，后端仍会按资料增强处理。
2. 对新前端保持严格一致：只要前端显式传 `use_document_scope=false`，后端就不会偷偷使用文档作用域。
3. 记忆 / 资料库能力本次没有删除，只是不再作为首页默认表达和默认行为暴露。
4. 本次主要是交互与链路语义收口，不涉及数据库迁移。
