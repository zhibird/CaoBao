# v0.15.0 - 2026-04-12

## 背景与目标

`v0.15.0` 承接了项目从本地原型向更正式后端系统演进的需求。

在此前版本中，系统虽然已经引入 Alembic 并具备持续演进 schema 的能力，但默认运行时数据库仍然偏向 `SQLite`，更适合单机原型和轻量本地开发，不适合作为长期服务端架构的主数据库。

这个版本的目标，就是把数据库主路径从 `SQLite` 切换到 `PostgreSQL`，让项目在配置、容器、启动校验和测试层面都真正进入更接近正式服务的数据库运行方式，同时保留 `SQLite` 作为历史备份和兼容调试路径。

## 版本概览

1. 项目默认主库正式切换为 `PostgreSQL`，不再隐式回落到 `SQLite`。
2. Docker Compose 新增 `postgres` 服务，应用容器默认通过 PostgreSQL 连接串启动。
3. 运行时 legacy bootstrap 被限制为只服务 `SQLite` 兼容场景，正式 schema 演进统一回到 Alembic。
4. 测试基建补齐 PostgreSQL 路径校验与迁移 smoke coverage，确保主链路不是“只在 SQLite 上通过”。

## 主要更新

### 1. 默认数据库主路径切换到 PostgreSQL

1. `DATABASE_URL` 改为必填配置，不再在运行时偷偷生成默认 `SQLite` 路径。
2. 新增 `psycopg[binary]`，让 SQLAlchemy 与 Alembic 可以直接连接 `PostgreSQL`。
3. `.env.example` 改为以 `PostgreSQL` 为默认示例配置，明确 `POSTGRES_DB / POSTGRES_USER / POSTGRES_PASSWORD` 与标准连接串写法。

### 2. 运行时初始化逻辑收束到正式数据库流程

1. 启动时继续保留 schema 校验能力，要求数据库位于 Alembic head。
2. 原有 legacy bootstrap 不再对非 `SQLite` 数据库生效，避免在 `PostgreSQL` 上走临时补洞式建表逻辑。
3. `SQLite` 兼容逻辑仍然保留，但角色已经降级为历史数据兼容和本地调试兜底，不再是默认主路径。

### 3. Compose 与本地环境改为 PostgreSQL-first

1. `docker-compose.yml` 新增 `postgres` 服务、数据卷与健康检查。
2. `caibao-api` 默认依赖 `postgres` 就绪后启动，并通过容器内 PostgreSQL 连接串访问主库。
3. `Dockerfile` 移除对默认 `SQLite` 数据库路径的依赖，容器运行形态更贴近真实服务端部署。

### 4. 测试与迁移验证补齐 PostgreSQL 主链路

1. 新增配置测试，锁定“必须显式提供 `DATABASE_URL`”这一行为。
2. 新增数据库会话测试，校验 `SQLite` 与 `PostgreSQL` 在 legacy init 策略上的分流边界。
3. 测试基建支持通过 `TEST_DATABASE_URL` 直接跑 PostgreSQL。
4. 迁移测试补充 PostgreSQL schema 级 smoke test，验证 Alembic 升级到 head 后关键表是否完整落地。

## 兼容性说明

1. 本次升级的是“主库运行路径”，不是“自动搬迁历史数据”；现有 `SQLite` 数据不会自动导入 `PostgreSQL`。
2. `SQLite` 仍可用于备份检查或非 Docker 本地调试，但不再作为默认运行数据库。
3. 正式环境应通过 `alembic upgrade head` 管理 schema，不再依赖 legacy create-all 初始化。

## 验证结果

1. `pytest -q`，结果为 `110 passed, 1 skipped`。
2. PostgreSQL 路径的测试基建与迁移 smoke coverage 已补齐。
3. 当前环境未执行 Docker 实机联调；Compose 配置已完成，容器链路需在具备 Docker 的环境中进一步验证。
