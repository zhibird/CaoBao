# Step 11: LLM 输出治理设计

## 文档目标

本文档用于定义 `v0.10.4` 之后的下一阶段设计方案，目标是解决“大模型回答过短、被截断、续写体验不自然”这类问题，同时保持企业级软件要求的稳定性、可控性和成本边界。

这是一份设计提案，不是当前已落地实现。

## 背景

在 `v0.10.4` 之前，系统对 LLM 请求使用固定的 `max_tokens=512`，且未处理 `finish_reason=length`。这会导致：

1. 模型实际已经被截断
2. 后端仍将截断结果当成完整回答返回
3. 前端只能显示不完整内容

`v0.10.4` 已通过“自动续写补齐”修复主问题，但这仍属于兜底型方案，不应该是长期主路径。

## 核心判断

### 1. 不建议完全取消输出上限

企业级产品不应采用“无限输出”策略，原因如下：

1. 多数模型接口并不存在真正意义上的无限输出
2. 不设置上限会导致单次请求成本、时延和资源占用不可控
3. 模型在长输出场景下更容易出现重复、跑偏或循环
4. 服务端必须具备对异常输出的刹车能力
5. 审计、超时、并发、计费都依赖明确的上限治理

### 2. 建议采用企业级输出治理

更合理的目标不是“去掉限制”，而是：

1. 让默认输出上限足够高
2. 让上限根据模型能力和输入长度动态计算
3. 让截断变成可检测、可提示、可继续的产品行为
4. 让自动续写只作为有限兜底，而不是默认主链路

这更接近豆包、GPT 一类产品的工程实践。

## 设计目标

### 业务目标

1. 降低用户感知到的“回答被截断”
2. 对长回答场景提供稳定、连续、可继续的体验
3. 控制模型调用成本和平均响应时间
4. 为不同模型提供一致但可配置的行为

### 技术目标

1. 输出预算可预测
2. 截断状态可观测
3. 续写行为可配置
4. 前后端状态语义一致

### 非目标

1. 不在本阶段引入真正的全文流式断点恢复
2. 不在本阶段做按段落级别的增量缓存拼接
3. 不在本阶段接入复杂的 tokenizer 供应商适配层

## 总体方案

### 主原则

采用以下组合策略：

1. 高默认上限
2. 动态输出预算
3. 有限自动续写
4. 用户可见的继续生成能力
5. 完整的观测与审计

### 推荐默认行为

1. 默认 `LLM_MAX_TOKENS` 提高到 `4096`
2. 每次请求根据模型上下文窗口动态再裁剪一次
3. 如果回答命中长度上限：
   - 后端最多自动续写 1 次
   - 若仍未结束，前端展示“继续生成”按钮
4. 若模型正常结束，则不触发任何补救动作

## 详细设计

### 一、后端预算计算

#### 1. 输入参数

服务端在发起请求前，需要具备以下输入：

1. 模型名
2. 模型最大上下文窗口
3. 模型最大单次输出上限
4. 当前消息总输入 token 估算值
5. 系统保留安全余量
6. 产品级最大输出上限

#### 2. 建议公式

可输出预算建议按以下思路计算：

`response_budget = min(product_cap, model_output_cap, model_context_window - input_tokens - safety_margin)`

再对结果做最小值保护：

`response_budget = max(response_budget, minimum_answer_budget)`

#### 3. 推荐配置项

建议新增以下配置项：

1. `LLM_MAX_TOKENS_DEFAULT=4096`
2. `LLM_MAX_TOKENS_MIN=512`
3. `LLM_OUTPUT_SAFETY_MARGIN=1024`
4. `LLM_AUTO_CONTINUE_MAX_ROUNDS=1`
5. `LLM_AUTO_CONTINUE_ENABLED=true`
6. `LLM_CONTINUE_BUTTON_ENABLED=true`

#### 4. 模型能力表

建议维护一份模型能力注册表，至少包含：

1. `context_window`
2. `max_output_tokens`
3. `supports_vision`
4. `supports_streaming`

当用户选择模型后，后端优先使用注册表能力；如果没有登记，再退回全局默认值。

### 二、后端截断处理

#### 1. finish reason 处理规则

当模型返回结果后，后端必须记录 `finish_reason`：

1. `stop`
   - 正常结束
2. `length`
   - 达到输出预算，需要触发截断治理
3. `content_filter`
   - 内容策略拦截
4. 其他
   - 记录并按错误或异常态处理

#### 2. 自动续写策略

自动续写不应再作为默认主策略，只作为兜底：

1. 仅在 `finish_reason=length` 时触发
2. 默认最多续写 1 次
3. 续写前将上一段 assistant 内容加入上下文
4. 使用固定续写提示词，要求“从中断处继续，不重复前文”
5. 如果续写后仍然 `length`，则停止并向前端返回“可继续生成”的状态

#### 3. 返回结构建议

建议聊天接口在响应中增加输出治理元信息，例如：

```json
{
  "answer": "....",
  "finish_reason": "length",
  "is_truncated": true,
  "auto_continue_rounds": 1,
  "can_continue": true,
  "budget": {
    "input_tokens_estimate": 6200,
    "output_budget": 4096,
    "safety_margin": 1024
  }
}
```

这样前端可以明确知道当前是“正常完成”还是“达到上限后的阶段性结果”。

### 三、前端交互设计

#### 1. 基础展示

当前主消息区域无需做长度裁剪，保持完整展示。

在此基础上，增加状态提示：

1. 正常结束时不显示额外标记
2. 若 `is_truncated=true`，在消息尾部显示轻量提示：
   - “已达到本次输出上限”

#### 2. 继续生成按钮

当 `can_continue=true` 时，展示“继续生成”按钮：

1. 点击后发起 continuation 请求
2. continuation 请求携带当前会话消息和上次回答尾部
3. 新结果追加到同一条 assistant 消息下，而不是创建一条全新消息

#### 3. 自动与手动的边界

建议体验为：

1. 后端自动续写 1 次，用户通常无感知
2. 若仍不足，再由用户手动点击“继续生成”
3. 避免系统连续多轮自动追问导致时延和费用不可控

### 四、接口设计建议

#### 1. 现有聊天接口

保留当前 `/chat/ask` 主接口，但在响应里增加：

1. `finish_reason`
2. `is_truncated`
3. `can_continue`
4. `auto_continue_rounds`
5. `budget`

#### 2. 新增继续生成接口

建议新增：

`POST /api/v1/chat/continue`

请求参数建议：

```json
{
  "conversation_id": "xxx",
  "assistant_message_id": "xxx",
  "team_id": "team1",
  "user_id": "user1"
}
```

响应结构与 `/chat/ask` 一致，但语义是“续接上一次回答”。

### 五、可观测性与审计

#### 1. 建议埋点字段

每次 LLM 调用建议记录：

1. `conversation_id`
2. `model`
3. `input_tokens_estimate`
4. `output_budget`
5. `finish_reason`
6. `auto_continue_rounds`
7. `latency_ms`
8. `is_truncated`

#### 2. 关键指标

建议关注：

1. 截断率
2. 自动续写触发率
3. 用户手动继续生成点击率
4. 平均响应时延
5. 平均输出 token
6. 单请求成本

### 六、异常与边界场景

#### 1. 极长上下文

如果输入已经接近模型上下文上限：

1. 后端不应盲目继续压缩输出预算到极低值
2. 当预算低于最小阈值时，应优先压缩输入上下文或减少检索片段
3. 必要时返回明确提示：
   - “上下文过长，本次回答可能简化”

#### 2. 模型能力未知

如果当前模型没有登记上下文窗口和最大输出能力：

1. 使用全局默认预算
2. 记录警告日志
3. 在后续运维中补齐该模型能力元数据

#### 3. 续写失败

如果自动续写失败：

1. 保留已生成内容
2. 将 `can_continue` 设为 `true`
3. 将错误写入日志
4. 前端允许用户稍后重试

## 数据结构建议

### 后端响应 Schema 扩展

建议为聊天响应增加字段：

1. `finish_reason: string | null`
2. `is_truncated: bool`
3. `can_continue: bool`
4. `auto_continue_rounds: int`
5. `budget_meta: dict`

### 会话消息存储建议

如果未来要支持更稳定的“继续生成”，建议在消息表或扩展字段中存：

1. `finish_reason`
2. `is_truncated`
3. `generation_round`
4. `parent_message_id`

这样前端刷新后仍能知道某条消息是否为“尚可继续”的阶段性结果。

## 测试方案

### 单元测试

至少覆盖：

1. 正常 `stop` 返回
2. `length` 后自动续写 1 次成功
3. `length` 后自动续写仍被截断
4. 预算计算结果低于最小阈值
5. 未知模型能力下的默认预算回退

### 集成测试

至少覆盖：

1. 长问题 + 多检索片段 + 高输出预算
2. 长问题 + 图片附件 + 视觉模型
3. 长问题 + OCR fallback
4. 用户点击“继续生成”后的消息拼接

### 前端测试

至少覆盖：

1. 正常消息无截断提示
2. 截断消息显示“已达到本次输出上限”
3. `can_continue=true` 时出现“继续生成”按钮
4. 点击继续后内容被追加而不是新建独立消息

## 发布建议

### 阶段 1

1. 保留现有自动续写机制
2. 将自动续写轮数降到 1
3. 响应里补充 `finish_reason/is_truncated/can_continue`

### 阶段 2

1. 引入动态输出预算
2. 为模型维护能力注册表
3. 前端接入“继续生成”按钮

### 阶段 3

1. 打通埋点与审计面板
2. 根据真实数据优化预算默认值
3. 评估是否需要更精细的 tokenizer 估算

## 最终建议

推荐方向不是“彻底取消输出上限”，而是：

1. 用更高的默认上限降低用户感知
2. 用动态预算保证稳定性和成本可控
3. 用有限自动续写兜底
4. 用“继续生成”把剩余控制权交给用户

这条路线更符合企业级软件对体验、成本、稳定性和可观测性的综合要求。
